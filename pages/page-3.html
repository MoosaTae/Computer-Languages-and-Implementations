Context-Free Grammars & Left Recursion

The grammar IS the language. Get this wrong and your parser loops forever.

The three things exams always test: (1) identify left recursion, (2) eliminate it mechanically, (3) spot ambiguity and fix it with grammar hierarchy. Everything else here supports those three skills.

CFG: The Four Components

A Context-Free Grammar is a 4-tuple: terminals, non-terminals, productions, and a start symbol. Every structured language you have ever used is defined by one.

ComponentWhat it isExamples
TerminalActual tokens from the scanner. Cannot be expanded.id, +, (, 42, if
Non-terminalAbstract syntactic categories. Must be expanded by a production.Expr, Term, Stmt, Factor
ProductionA rewriting rule: one non-terminal expands to a sequence of symbols.Expr -> Expr + Term | Term
Start symbolThe top-level non-terminal. A complete parse must reduce to this.Usually the first non-terminal listed
 
Expr   -> Expr + Term | Expr - Term | Term
Term   -> Term * Factor | Term / Factor | Factor
Factor -> id | num | ( Expr )
Terminals come from the scanner. Non-terminals exist only in the grammar. A parse tree starts at the start symbol and has terminals as every leaf.

BNF vs EBNF Notation

BNF (Backus-Naur Form) is the original notation. EBNF adds two shorthand operators that map directly to control flow in parser code -- this is why EBNF is what you actually write parsers from.

NotationSymbolMeaningCompiles to
BNF / EBNF::= or ->Production: "expands to"Function definition
BNF / EBNF|Alternation: choose one branchif / else if / else
EBNF only{ ... }Zero or more repetitionswhile loop
EBNF only[ ... ]Optional: zero or one occurrenceif statement (no else)
BNF convention<E>Non-terminal (angle-bracket style)--
 
BNF style:
  <Stmt> ::= <IfStmt> | <WhileStmt> | <Assign>
  <IfStmt> ::= if ( <Expr> ) <Stmt> | if ( <Expr> ) <Stmt> else <Stmt>

EBNF style (more concise):
  Stmt   = IfStmt | WhileStmt | Assign
  IfStmt = if ( Expr ) Stmt [ else Stmt ]
  While  = while ( Expr ) Stmt
  Seq    = Stmt { ; Stmt }
The { } and [ ] in EBNF are not just syntactic sugar -- they eliminate left recursion from the grammar representation entirely. A { } is a while loop. A [ ] is an if with no else. Write EBNF and your recursive descent parser writes itself.

Left Recursion: Why It Kills Top-Down Parsers

A grammar is directly left-recursive when a non-terminal appears as the first symbol of its own production's right-hand side.

ex = ex op term | term    <-- left-recursive: ex starts by calling ex

Now trace what a recursive descent parser does:

call parseExpr() → first alt: call parseExpr() → first alt: call parseExpr() → ... infinite loop
The parser never consumes a single token before recursing. It has no base case to stop. The call stack blows up. Left recursion = stack overflow = wrong grammar for top-down parsing. This is not a performance issue -- it is a correctness issue.

The Mechanical Fix: Direct Left Recursion Elimination

The pattern is always the same. Identify it, apply the formula.

GENERAL PATTERN:

  Before: A = A a1 | A a2 | ... | b1 | b2 | ...
  After:  A = (b1 | b2 | ...) { a1 | a2 | ... }

SIMPLEST CASE:

  Before: A = A x | y
  After:  A = y { x }

The key insight: the b-alternatives (those that do NOT start with A) become the required prefix. The a-alternatives (those that DID start with A) become zero-or-more repetitions.

Full Example Walkthrough

Original grammar (all three rules are left-recursive):

  ex   = ex + term | ex - term | term
  term = term * factor | factor
  factor = ( ex ) | id | num

After elimination:

  ex   = term { (+ | -) term }
  term = factor { * factor }
  factor = ( ex ) | id | num
BeforeAfterMaps to code
ex = ex op term | termex = term {op term}while loop on op
term = term * f | fterm = f {* f}while loop on *
factor = ( ex ) | id | numunchanged (no left recursion)if/else if on lookahead
The rewritten grammar generates the same language as the original. Elimination does NOT change what strings are in the language -- it only changes the grammar's structure to be safe for top-down parsing.

Indirect Left Recursion

Left recursion can be hidden: A derives itself through a chain of other non-terminals.

Direct:   A -> A alpha | beta          (obvious)

Indirect: A -> B c
          B -> A d
          A => B c => A d c            (A derives A through B)

The general notation: A =>+ A Y means A derives something starting with A in one or more steps.

The full algorithm for indirect left recursion (Paull's algorithm) involves substituting productions in a specific ordering. However, as noted in this course: indirect left recursion essentially never appears in real programming language grammars. For this exam, mastering direct left recursion removal is sufficient. If you see it on a problem, the fix is still: find the cycle, substitute to make it direct, then apply the standard formula.

Ambiguous Grammars

A grammar is ambiguous if there exists at least one string with two different parse trees. Two parse trees = two different meanings = the language is not well-defined.

Example 1: Expression Ambiguity

Ambiguous grammar:
  E -> E + E | E * E | id

String: id + id * id

Parse tree 1 (+ at root):         Parse tree 2 (* at root):
        +                                  *
       / \                                / \
      id  *                              +   id
         / \                            / \
        id  id                         id  id

These give different values: (a + b) * c  vs  a + (b * c)

Example 2: The Dangling Else

Grammar:
  Stmt -> if ( Expr ) Stmt
        | if ( Expr ) Stmt else Stmt
        | other

String: if (a) if (b) s1 else s2

Two parses:
  1. else belongs to inner if:  if (a) { if (b) s1 else s2 }
  2. else belongs to outer if:  if (a) { if (b) s1 } else s2

Most languages resolve dangling else with the "match to nearest unmatched if" rule, implemented by splitting into two non-terminals:

MatchedStmt   = if ( Expr ) MatchedStmt else MatchedStmt | other
UnmatchedStmt = if ( Expr ) Stmt
              | if ( Expr ) MatchedStmt else UnmatchedStmt
Stmt          = MatchedStmt | UnmatchedStmt

Resolving Ambiguity: The Grammar Hierarchy Pattern

ProblemRoot CauseFix
PrecedenceGrammar has no notion of which op binds tighterPut higher-precedence ops deeper in the grammar (further from start symbol)
AssociativityGrammar allows grouping from either directionLeft-assoc: use { } in EBNF. Right-assoc: use right recursion (A -> B x A)
Dangling elseOne else can attach to multiple ifsSplit grammar into MatchedStmt / UnmatchedStmt
 
Unambiguous expression grammar (by hierarchy):

  expr   = term { (+ | -) term }       <-- lowest precedence, top level
  term   = factor { (* | /) factor }   <-- higher precedence
  factor = base { ^ base }             <-- even higher (exponentiation)
  base   = id | num | ( expr )         <-- highest: atoms and grouping

Rule: lower precedence op = higher in grammar = further from leaves.

Derivations: Leftmost vs Rightmost

A derivation is a sequence of rewriting steps from the start symbol to a terminal string. At each step you replace one non-terminal using one of its productions.

TypeWhich non-terminal to expandUsed by
Leftmost derivationAlways expand the leftmost non-terminal in the current sentential formTop-down parsers (LL)
Rightmost derivationAlways expand the rightmost non-terminal in the current sentential formBottom-up parsers (LR) -- in reverse
 
Grammar:  E -> E + T | T,  T -> T * F | F,  F -> id

Input: id + id * id

Leftmost derivation:
  E
  => E + T           (expand E, leftmost)
  => T + T           (expand E, leftmost)
  => F + T           (expand T, leftmost)
  => id + T          (expand F, leftmost)
  => id + T * F      (expand T, leftmost)
  => id + F * F      (expand T, leftmost)
  => id + id * F     (expand F, leftmost)
  => id + id * id    (expand F, leftmost)
For an unambiguous grammar, leftmost and rightmost derivations produce the same parse tree -- they just build it in different orders. For an ambiguous grammar, the same string can have two different leftmost derivations, which is exactly how you prove ambiguity.

Exam Checklist

  • Name the four components of a CFG: terminals, non-terminals, productions, start symbol
  • Convert between BNF and EBNF -- know that { } = while loop and [ ] = if statement
  • Identify direct left recursion in any grammar rule
  • Apply the elimination formula: A = A x | y becomes A = y { x }
  • Explain why left recursion causes infinite loops in recursive descent
  • Recognize indirect left recursion (A derives A through other non-terminals)
  • Prove ambiguity by drawing two distinct parse trees for one string
  • Resolve expression ambiguity using grammar hierarchy for precedence and associativity
  • Explain the dangling else problem and its grammar-level fix
  • Perform a leftmost derivation step-by-step and identify which parser type uses it
