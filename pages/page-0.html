Compiler Construction

Final Exam Survival Guide -- The 20% of concepts that cover 80% of exam power

A compiler is one thing: a function that maps source text to target instructions through a series of meaning-preserving transformations. Every topic in this course is a deep dive into one stage of that pipeline. Internalize the pipeline first, and everything else finds its place.

The Compilation Pipeline

Source code enters one end. Machine instructions exit the other. Six phases transform it step by step, each phase consuming the output of the previous one.

Source Text → Scanner → Parser → Semantic Analyzer → IR Gen → Optimizer → Code Gen → Target Code
Phase Input Output Core Task Exam Weight
Scanner Character stream Token stream Regex → NFA → DFA; longest match MOD
Parser Token stream Parse tree Grammar, FIRST/FOLLOW, LL(1) table, recursive descent HIGH
Semantic Analyzer Parse tree Annotated tree Type checking, scope resolution, symbol table MOD
IR Generator Annotated tree IR code (S-code) Tree walk emitting instructions; same structure as interpreter HIGH
Optimizer IR code Optimized IR Dead code, constant folding, loop transforms MOD
Code Generator Optimized IR Target instructions Register allocation, instruction selection, activation records HIGH

The 6 Pareto Concepts

This course has ten topics. Six of them are tested on nearly every exam. Master these six and you have covered the overwhelming majority of what can be asked.

1. The Compilation Pipeline

The six-phase model is the skeleton of the entire course. Every other topic hangs off one bone of this skeleton. If an exam question feels unfamiliar, ask: which phase does this belong to? That question alone will orient you and surface the right tools.

What to know cold: Phase names, input/output at each boundary, which phases are front-end (machine-independent) vs back-end (machine-specific).

2. FIRST / FOLLOW + LL(1) Parse Table

The mathematical engine behind top-down parsing. FIRST tells you which terminal can start a string; FOLLOW tells you what can legally come after a non-terminal when it derives empty. Together they fill the parse table cells. Conflicts in the table mean the grammar is not LL(1).

Algorithm: Initialize all sets to empty. Apply rules. Repeat until no set changes (fixed-point). Then fill table: for each production A → alpha, add it to M[A, t] for each t in FIRST(alpha); if epsilon in FIRST(alpha), also add to M[A, t] for each t in FOLLOW(A).

3. Grammar → Parser → Code

The grammar IS the structure of the parser. Remove left recursion first. Then translate mechanically: one function per non-terminal, match() calls for terminals, mutual recursion mirrors the productions. This is the core implementation skill and is tested heavily.

Left recursion removal: A → A alpha | beta becomes A → beta A' and A' → alpha A' | epsilon, or equivalently in EBNF: A → beta { alpha }.

4. S-code Generation

S-code is the virtual machine instruction set used in this course. The code generator walks the parse tree and emits instructions. This is structurally identical to an interpreter -- it just prints instructions instead of executing them.

Core instruction set: lit n (push constant), get/put (read/write local by offset), ld/st (read/write global), jf label (jump if false), jmp label (unconditional jump), call/rtn. Forward jumps require patching.

5. Activation Records + 4-Phase Linkage

Every function call creates an activation record (stack frame) at runtime. The AR holds everything the function needs to execute and to return correctly. The 4-phase linkage protocol (pre-call, prolog, epilog, post-return) is the choreography for building and tearing down frames.

AR fields in order: arguments, return value, return address, control link (caller's frame), access link (enclosing scope's frame), saved registers, local variables, temporaries. Access links, not control links, are used to resolve variable references across lexical scopes.

6. Two-Stack Parse Tree Construction

Building a parse tree during parsing uses two stacks: an argument (operand) stack for subtrees and an operator stack for pending operators. When operator precedence allows, makebintree() pops two subtrees and one operator and builds a new node. Action routines embedded in the parser call these operations at the right moments.

Why it matters: This is the bridge between parsing and code generation. The parse tree produced here is exactly what the S-code generator walks.

Quick-Reference Decision Table

When you see an exam question, map it to a concept immediately. Do not re-derive from scratch.

Exam question contains... Immediately think... Page
"Name the compiler phases" or "what is a compiler?" 6-phase pipeline, compiler vs interpreter distinction Page 1
"Token, lexeme, pattern" or "longest match" Scanner phase; token=category, lexeme=string, pattern=regex Page 2
Grammar with left recursion Rewrite using EBNF { } repetition or A' auxiliary non-terminal Page 3
"Compute FIRST" or "compute FOLLOW" Iterative fixed-point: start empty, apply all rules, repeat until stable Page 4
"Build the LL(1) parse table" FIRST fills cells; epsilon productions use FOLLOW; conflicts = not LL(1) Page 4
"Is this grammar LL(1)?" or "check for conflicts" Look for left recursion, common prefixes, multiple cells filled Page 4
"Write a recursive descent parser for..." One function per non-terminal, match() for terminals, lookahead Page 5
"Build the parse tree for..." or "trace the two stacks" Argument stack + operator stack, makebintree(), action routines Page 6
"Generate S-code for..." or "translate to S-code" get/put locals, ld/st globals, lit constants, jf/jmp control, patch forward jumps Page 7
"Show the activation record" or "trace the stack" Draw frame: args, ret val, ret addr, control link, access link, locals, temps Page 8
"How does a function call work?" or "4-phase linkage" Pre-call → Prolog → [execute body] → Epilog → Post-return Page 8
"Access variable from outer scope" or "static coordinate" Follow access links (not control links), use <level, offset> coordinate Page 8
"Mark-sweep" or "garbage collection" or "reference counting" Mark-sweep: 4 phases (alloc, mark roots, trace, sweep); ref counting: cycles are the problem Page 9

Study Priority Order

Not all topics are equal. Work through these in order. The first five alone cover the majority of a typical exam.

  • 1. S-code Generation (Page 7) -- Highest exam density. Practice translating 3-4 complete programs end-to-end. If-else, while loops, function calls, recursive calls.
  • 2. FIRST / FOLLOW + LL(1) Table (Page 4) -- Pure algorithm. Practice the fixed-point iteration on 2-3 grammars from scratch until it feels mechanical.
  • 3. Activation Records + Procedure Linkage (Page 8) -- Draw the full stack for a recursive call like fact(3). Label every field. Trace all 4 linkage phases.
  • 4. Recursive Descent Parser (Page 5) -- Start from a grammar, remove left recursion, write the parser functions. Do this without notes until it is automatic.
  • 5. Two-Stack Parse Tree Construction (Page 6) -- Trace makebintree() step-by-step on a 4-5 token expression. Know when each stack operation fires.
  • 6. CFG: Left Recursion + Ambiguity (Page 3) -- Identify and remove left recursion. Recognize ambiguity in if-else and arithmetic grammars.
  • 7. Compiler Phases -- Conceptual (Page 1) -- Quick review. Know the input/output of every phase. Know compiler vs interpreter distinction.
  • 8. Scanner (Page 2) -- Tokens, lexemes, patterns. Regex to NFA to DFA pipeline. Longest-match rule. Conceptual rather than deep algorithmic.
  • 9. Garbage Collection (Page 9) -- Mark-sweep four phases, reference counting and its weakness (cycles), generational GC idea. One question, not more.
  • 10. Optimizer (covered in Page 1) -- Know the categories: dead code elimination, constant folding, loop-invariant hoisting. No deep derivations expected.
The most common exam mistake is spending too long on Scanner/Regex questions (low weight) and running out of time for S-code and Activation Records (high weight). Budget time proportional to the weights above, not proportional to how familiar a topic feels.
The three heaviest exam topics -- S-code Generation, FIRST/FOLLOW, and Activation Records -- share a common structure: they are all mechanical algorithms with precise, reproducible steps. There is no insight required once you have internalized the procedure. The exam rewards practice, not cleverness.
