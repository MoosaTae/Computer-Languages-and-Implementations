Activation Records & Procedure Linkage

How the runtime manages function calls, local variables, and the contract between caller and callee.

Every function call in any compiled language — C, Java, Pascal, Go — relies on the same fundamental mechanism: an activation record on a stack, managed by a strict 4-phase protocol. Master this and you understand how ALL procedure calls work at the machine level.

What Is an Activation Record?

An activation record (AR), also called a stack frame, is a contiguous block of memory allocated each time a function is called. It contains everything needed to execute that invocation and return cleanly to the caller: arguments, local variables, saved registers, and the bookkeeping links that hold the call chain together.

One activation record per live call, not per function definition. If fact calls itself 3 times, there are 3 ARs on the stack simultaneously — each with its own copy of n.

AR Structure (Top to Bottom)

+---------------------------+  <-- high address (caller's territory)
|  Return value             |  result slot — caller reads this after return
+---------------------------+
|  Actual parameters        |  arguments passed by caller
+---------------------------+
|  Control link             |  pointer to caller's AR (dynamic chain)
+---------------------------+
|  Access link              |  pointer to lexically enclosing AR (static chain)
+---------------------------+
|  Saved machine status     |  return address, saved registers
+---------------------------+
|  Local variables          |  this function's locals
+---------------------------+
|  Temporaries              |  intermediate expression values
+---------------------------+  <-- low address (stack grows down)

x86-64 Stack Frame Layout

On x86-64, two registers anchor every frame. %rbp (frame pointer) is a fixed reference — it does not move while the function runs. %rsp (stack pointer) marks the current top of the stack and shifts as data is pushed or popped. 

[Higher addresses]
  ...caller's frame...
  ----------------------------------------
  arg n          [8n+8(%rbp)]   <-- nth argument
  ...
  arg 2          [16(%rbp)]
  arg 1          [8(%rbp)]      <-- first argument  (NOTE: some ABIs use registers)
  return address [8(%rbp)]      <-- pushed by CALL instruction
  saved %rbp     [0(%rbp)]      <-- old frame pointer stored here  <-- %rbp points here
  ----------------------------------------
  local 1        [-8(%rbp)]
  local 2        [-16(%rbp)]
  ...
  local m        [-8m(%rbp)]    <-- %rsp points here (or lower)
[Lower addresses]
Address Contents Who owns it
8n+8(%rbp) ... 16(%rbp) Actual arguments (arg n down to arg 1) Placed by caller before call
8(%rbp) Return address (pushed by CALL) CPU / caller
0(%rbp) Saved old %rbp (caller's frame pointer) Callee saves in prolog
-8(%rbp) ... -8m(%rbp) Local variables Callee allocates in prolog
The modern x86-64 System V ABI passes the first 6 integer arguments in registers (%rdi, %rsi, %rdx, %rcx, %r8, %r9), not on the stack. Only arguments beyond the 6th spill to the stack. Know which convention your exam uses — the conceptual layout above is what matters for understanding the structure.

The 4-Phase Procedure Linkage Protocol

Every function call is a cooperative handoff. Caller and callee each have defined responsibilities split across four phases. Violate the protocol and the stack corrupts.

Pre-call (Caller) → Prolog (Callee) → Epilog (Callee) → Post-return (Caller)

Phase 1: Pre-call (done by CALLER)

StepPurpose
Allocate space for callee's ARMake room on the stack for the frame
Store arguments in parameter sectionPass values the callee will read
Save return addressCALL instruction pushes %rip automatically
Save caller-save registersPreserve live values that callee may overwrite
Jump to callee's prologTransfer control (CALL instruction)

Phase 2: Prolog (done by CALLEE)

StepPurpose
Save old frame pointer (push %rbp)Remember caller's frame reference
Set new frame pointer (mov %rsp, %rbp)Establish stable reference for this frame
Save callee-save registersPreserve registers this function will overwrite
Allocate locals (sub $N, %rsp)Extend the AR for local variables and temporaries

Phase 3: Epilog (done by CALLEE)

StepPurpose
Store return value (into %rax)Put result where caller expects it
Restore callee-save registersUndo what prolog saved
Restore caller's frame pointer (pop %rbp)Point %rbp back to caller's frame
RET instructionPop return address and jump back to caller

Phase 4: Post-return (done by CALLER)

StepPurpose
Copy return value from %raxGet the result into the right location
Free callee's AR (add $N, %rsp)Reclaim stack space
Restore caller-save registersRecover saved values
Continue executionResume after the call site

Register Saving Convention

CategoryWho savesRationale
Caller-save registers Caller (before call) Only the caller knows which values are live across the call
Callee-save registers Callee (in prolog) Callee knows which registers it will actually use
Reserved (%rbp, return addr) Neither — convention-managed Handled structurally by the linkage protocol itself

Access Link vs. Control Link

This is one of the most commonly confused distinctions in the topic.

Control Link (Dynamic) Access Link (Static)
Points to The AR of whoever called this function The AR of the lexically enclosing procedure
Follows The dynamic call chain (runtime execution order) The static nesting structure (source code layout)
Used for Returning to the caller after the function ends Accessing variables from outer lexical scopes
Same as caller? Always — by definition Only when called from the lexically enclosing scope
Control link answers: "Who called me?" Access link answers: "Where were my enclosing variables defined?" These are only the same in simple, non-nested cases. In languages with nested procedures (Pascal, Ada, some functional languages), they diverge.

When They Differ: A Concrete Example

procedure main {
  int x;                    -- declared at level 0

  procedure p1 {            -- nested in main, level 1
    use x;                  -- accesses main's x
  }

  procedure p2 {            -- nested in main, level 1
    procedure q2 {          -- nested in p2, level 2
      call p1;              -- p1 is a sibling in main's scope
    }
    call q2;
  }

  call p2;
}

When p1 is called from inside q2:

  • Control link of p1's AR points to q2's AR (that's who called it at runtime)
  • Access link of p1's AR points to main's AR (that's where p1 is textually defined, so that's where its non-local variables live)
The access link is set by the CALLER, not the callee. The caller knows the static nesting context and must pass the right enclosing scope pointer. The callee cannot figure this out on its own.

Static Coordinates and Variable Access

The compiler translates every variable reference into a static coordinate <level, offset>:

  • level = lexical nesting depth of the scope where the variable is declared
  • offset = byte offset within that scope's local data area

At runtime, to access <level, offset> from within a procedure at nesting level k, follow the access link chain (k - level) times, then add the offset.

Static Coordinate Hops needed (current level = 2) Access code (access link at ARP-4)
<2, 8> — same level 0 hops — use current ARP directly loadAI r0, 8 => r10
<1, 12> — one level out 1 hop — follow access link once loadAI r0, -4 => r1 ; loadAI r1, 12 => r10
<0, 16> — two levels out 2 hops — follow access link twice loadAI r0, -4 => r1 ; loadAI r1, -4 => r1 ; loadAI r1, 16 => r10
The cost of non-local variable access is proportional to lexical distance. Deeply nested access becomes expensive. This is why most practical languages (C, Go) forbid nested function definitions — it eliminates access links entirely and makes all non-local access O(1) via the static data area.

Recursive Call Example: fact(3)

fact(n) = if n <= 1 then 1 else n * fact(n-1)

Calling fact(3) builds this stack (growing downward):

+-------------------------------+  <-- bottom of call chain
| AR: fact(3)                   |
|   n = 3                       |
|   control link ---------------+--> caller (main)
|   return-result slot          |
+-------------------------------+
        |  calls fact(2)
        v
+-------------------------------+
| AR: fact(2)                   |
|   n = 2                       |
|   control link ---------------+--> fact(3)'s AR
|   return-result slot          |
+-------------------------------+
        |  calls fact(1)
        v
+-------------------------------+
| AR: fact(1)                   |
|   n = 1                       |
|   control link ---------------+--> fact(2)'s AR
|   return-result slot = 1      |
+-------------------------------+  <-- top of stack (deepest frame)

Unwinding: fact(1) returns 1 into its return slot. fact(2) reads that, computes 2 * 1 = 2, stores it, pops its own frame. fact(3) reads 2, computes 3 * 2 = 6. Each frame is independent — they share no variables.

Recursive calls work because each activation gets its own AR. The function definition is shared (one copy of code), but execution state is not. Stack depth = recursion depth, which is why deep recursion causes stack overflow.

Where Do Activation Records Live?

Scenario Where the AR lives Why
Normal functions (LIFO call pattern) Stack Cheap allocation/deallocation; last-in first-out matches call return order
Closures / functions that outlive their caller Heap The AR must persist after the creating function returns; stack would reclaim it
Functions with no recursive calls and no calls to other functions (leaf) Static data area Only one invocation ever active; no need for a dynamic stack slot
If a language supports first-class functions or closures (Python, JavaScript, Scheme, Rust closures), the compiler may need to "escape analyze" each variable. Variables that are captured by a closure that outlives the current scope must be allocated on the heap, not the stack. Stack-allocated closures are a common source of dangling-pointer bugs in C.

Runtime Memory Layout

At runtime, the process address space is divided into four regions. Stack and heap grow toward each other, sharing unused middle space efficiently.

+--------------------------+  high address
|         Code             |  -- read-only; fixed size; generated by compiler
+--------------------------+
|       Static Data        |  -- global variables, string literals; fixed size
+--------------------------+
|         Stack            |  -- activation records; grows DOWNWARD
|           |              |
|           v              |
|                          |  -- unused address space in the middle
|           ^              |
|           |              |
|          Heap            |  -- malloc/new; grows UPWARD
+--------------------------+  low address
Variable typeWhere it lives
Local variables & parametersActivation record on the stack
Static / global variablesStatic or global data area
Heap-allocated objects (malloc, new)Heap
Variable-length arraysDescriptor in AR; actual data at end of AR or on heap
Machine codeCode segment (text section)
