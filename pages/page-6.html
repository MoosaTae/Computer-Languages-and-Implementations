Parse Trees & Action Routines

The bridge between parsing and code generation — where syntax becomes structure.

Master cons cells, the two-stack algorithm, and makebintree(). These three concepts are the entire mechanism by which a recursive descent parser produces a tree that code generation can walk. Everything else is detail.

What Is a Parse Tree?

Parsing answers "is this valid syntax?" A parse tree answers "what does it mean?" It encodes operator precedence and grouping as structure, not text. The expression a + b * c becomes:

+ / \ a * / \ b c

The tree enforces that b * c is computed first — not by rules, but by shape. Code generation simply walks the tree depth-first: leaves are operands, internal nodes are operations.

Source text → Parser + Action Routines → Parse Tree → Code Generator → Output Code

Data Structure: Cons Cells

A parse tree is built from cons cells — the same linked-list primitive from Lisp. Every cell has two fields: head and tail.

There are exactly two kinds of cells:

TypeWhat it isAnalogy
Atom A leaf node. Holds a raw value: a variable name, a number, an operator symbol. A word in a sentence — has no internal structure.
Dot-pair An internal node. Holds two pointers: head points to one subtree, tail points to the rest of the list. A branch in a tree — connects things together.

Core Operations

OperationWhat it doesWhen you use it
isatom(a)Returns true if a is a leaf (atom)Base case of recursion
head(a)Returns the first element of a listGet the operator of a subtree
tail(a)Returns the rest of the list after headGet the operands of a subtree
cons(h, t)Creates a new dot-pair cell: head=h, tail=tBuilding nodes during parsing
newatom(x)Creates a new atom with value xTurning a token into a leaf node
The expression (+ a b) as a list means: head = +, tail = a list (a b). That tail is itself: head = a, tail = (b). Which is: head = b, tail = NIL. Every binary operation becomes a 3-element list in this prefix notation.

The recursive structure of cons cells mirrors the recursive structure of expressions. That is not a coincidence — it is why this representation works.

-- Copying a tree: the recursion perfectly matches the data structure
copy(a)
  if isatom(a)
    return newatom(copyof(a))       -- leaf: copy the value
  else
    return cons(copy(head(a)), copy(tail(a)))  -- branch: copy both sides

The Two-Stack Strategy

The parser cannot build a tree in a single pass over the token stream because it does not know the full context of an operator when it first sees it. The solution: use two stacks to defer decisions until you have all the pieces.

StackNameHoldsThink of it as
ystack Argument stack Operands and completed subtrees The "values" waiting to be combined
zstack Operator stack Operators pending application The "verbs" waiting for their arguments 
push(arg)      -- push operand or completed subtree onto ystack
pushop(op)     -- push operator onto zstack
pop()          -- pop from ystack (returns a subtree or atom)
popop()        -- pop from zstack (returns an operator)
Two stacks is the key architectural decision. The argument stack accumulates subtrees bottom-up. The operator stack keeps operators until their operands are ready. makebintree() is the moment they combine.

The Key Function: makebintree()

This is the single most important function on this page. It fires whenever a complete binary operation is ready: two operands on ystack, one operator on zstack.

makebintree()
  b  = pop()          -- right operand (top of ystack)
  a  = pop()          -- left operand
  op = popop()        -- operator from zstack
  c  = cons(op, cons(a, cons(b, NIL)))   -- build (op a b)
  push(c)             -- push the new subtree back onto ystack

The result is a 3-element list (op a b) in prefix notation, pushed back as a single unit. From the perspective of the surrounding expression, this subtree is now just another operand.

After makebintree(), the two operands and their operator disappear from the stacks and are replaced by a single subtree node. This is the tree being built bottom-up, incrementally, in-place.
Pop order matters. pop() is called twice: first pop is the right operand, second pop is the left operand. Reverse this and you build a mirrored tree. For commutative operators like + the result is the same, but for - and / it is silently wrong.

Action Routines Embedded in the Parser

An action routine is a call to a tree-building function inserted directly inside the grammar rules of the parser. The parser drives the recognition; the action routines build the structure as a side effect.

Grammar: ex = term { op term } and term = ( ex ) | id

op()
  if tok == +
    match(+)
    pushop(+)               -- ACTION: stash the operator for later
  else if tok == -
    match(-)
    pushop(-)               -- ACTION: stash the operator for later
  else if tok == *
    match(*)
    pushop(*)
  else if tok == /
    match(/)
    pushop(/)

term()
  if tok == (
    match( ( )
    ex()                    -- recursively parse the inner expression
    match( ) )
    makebintree()           -- ACTION: seal inner result into a subtree
  else if tok == id
    match(id)
    push(id)                -- ACTION: push leaf atom onto ystack

ex()
  term()
  if tok == + || tok == - || tok == * || tok == /
    op()
    term()
    makebintree()           -- ACTION: combine left op right into a subtree
Notice the pattern: every time a complete "unit" is recognized, an action routine fires. Recognizing an id? Push a leaf. Closing a parenthesized group? Seal it with makebintree(). Completing an op term sequence? Combine with makebintree(). The parser and the tree builder are synchronized.

Handling Parenthesized Expressions

Parentheses mean "treat everything inside me as a single operand." The two-stack approach handles this naturally because makebintree() is called immediately after the closing paren is matched — before the surrounding expression continues.

term()
  if tok == (
    match( ( )
    ex()                 -- everything inside gets pushed/popped within ex()
    match( ) )
    makebintree()        -- the inner result is now ONE subtree on ystack

Consider (a + b) * c:

  1. Match (, recurse into ex()
  2. Inside: push a, pushop +, push b
  3. Match ), call makebintree() — ystack now has (+ a b) as one item
  4. Back in outer ex(): pushop *, push c
  5. Call makebintree() — pops c, (+ a b), and *, builds (* (+ a b) c)
Without calling makebintree() at the close-paren, the inner operators and operands stay on the stacks and get mixed up with the outer expression. The close-paren call is what gives parentheses their binding power.

Complete Worked Example: a + b * c

Let us trace every step for the expression a + b * c. We assume * binds tighter by appearing as a nested term (or by grammar structure). In a flat grammar this trace shows how makebintree() fires after each complete operation.

Step-by-step trace

#TokenActionystack (arg)zstack (op)
1apush(a)[a][]
2+pushop(+)[a][+]
3bpush(b)[a, b][+]
4*pushop(*)[a, b][+, *]
5cpush(c)[a, b, c][+, *]
6(end of inner term)makebintree() for *[a, (* b c)][+]
7(end of ex)makebintree() for +[(+ a (* b c))][]

Resulting tree

+ / \ a * / \ b c

In list notation: (+ a (* b c)). The * node is nested inside the + node, encoding that multiplication happens first.

The order in which makebintree() fires determines the shape of the tree, which determines operator precedence. Grammar structure and action routine placement are the two levers for controlling this. This is exactly how operator precedence is encoded without any explicit precedence rules.

How Parse Trees Connect to Code Generation

Once the tree exists, code generation becomes a recursive tree walk. The tree is the only interface between the two phases — the parser does not need to know about registers, and the code generator does not need to know about grammar.

Leaf node (atom) → emit LOAD instruction
Internal node (op a b) → gen(a), gen(b), emit OP instruction 
gen(node)
  if isatom(node)
    emit("LOAD", node)           -- leaf: load the variable
  else
    gen(head(tail(node)))        -- generate left operand
    gen(head(tail(tail(node))))  -- generate right operand
    emit(head(node))             -- emit the operator as an instruction

This post-order traversal naturally produces the correct evaluation order: both operands are computed before the operation that uses them. No additional scheduling is needed.

  • I can trace ystack and zstack at each step for a given input expression.
  • I understand why makebintree() is called immediately after the closing paren in term().
  • I can explain how the parse tree structure encodes operator precedence without explicit precedence rules.
  • I can describe how gen(node) walks the parse tree to produce code.
