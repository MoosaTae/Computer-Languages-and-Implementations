Recursive Descent Parsing

One function per non-terminal. The grammar IS the code.

Master these three mappings and you can write a parser for any LL(1) grammar in minutes: sequence becomes sequential calls, alternation becomes if/else on the current token, repetition becomes a while loop.

The Core Idea

A recursive descent parser is a direct mechanical translation of a grammar into mutually recursive functions. Each non-terminal in the grammar becomes exactly one function in your code. The structure of the grammar is preserved perfectly in the call graph of the program.

Grammar rule → One function → Calls match() or other functions → Parse tree built by call stack

Each function does the same five things:

  1. Look at the current token (lookahead, does not consume)
  2. Decide which production applies based on FIRST sets
  3. For each terminal in that production: call match()
  4. For each non-terminal in that production: call the corresponding function
  5. For EBNF {...} repetition: loop with while

The match() Function

match() is the atomic operation of every recursive descent parser. It has one job: confirm the current token is what we expect, then advance the scanner.

match(expected):
  if tok == expected
    tok = next_token()    // advance -- consume the token
  else
    error("expected " + expected + ", got " + tok)
Lookahead PEEKS at the current token without consuming it. match() CONSUMES the token and moves forward. Never confuse these two. Lookahead is just reading tok. Consuming is calling match().

Translating Grammar Rules to Code

Every grammar construct has a direct, mechanical translation:

Grammar Construct Code Translation
A = X Y Z (sequence) X(); Y(); Z(); — sequential calls
A = X | Y (alternation) if (tok in FIRST(X)) X(); else if (tok in FIRST(Y)) Y();
A = {X} (EBNF repetition) while (tok in FIRST(X)) X();
Terminal t match(t);
Non-terminal B B(); — function call

Concrete Example: Arithmetic Expressions

Starting grammar:

ex   = term {op term}
op   = + | -
term = ( ex ) | id

Direct translation to parser functions:

op():
  if tok == '+'   match('+')
  else if tok == '-'  match('-')
  else  error("expected + or -")

term():
  if tok == '('       // alternation: peek to decide which production
    match('(')
    ex()
    match(')')
  else if tok == id
    match(id)
  else
    error("expected ( or id")

ex():
  term()              // first, always parse one term
  while tok == '+' || tok == '-'   // {op term} becomes while loop
    op()
    term()
The while loop is the key. EBNF {op term} means "zero or more repetitions of op term". That is precisely what while (tok in FIRST(op)) { op(); term(); } does. You loop as long as the next token starts a new repetition.

Handling Alternatives with Lookahead

When a non-terminal has multiple productions, you use the current token to decide which one to take. This is the "predictive" part of predictive parsing.

// Grammar: stmt = if-stmt | while-stmt | assign-stmt
stmt():
  if tok == IF
    if_stmt()
  else if tok == WHILE
    while_stmt()
  else if tok == ID
    assign_stmt()
  else
    error("expected if, while, or identifier")

The token you check against is exactly the FIRST set of each alternative. If you computed FIRST sets on the previous page, you already have the condition for every if/else branch.

Every decision point must have a final else that calls error. If you forget it, the parser silently does nothing when it encounters an unexpected token. Always close every if/else chain with an error case.

EBNF Repetition Becomes while Loops

EBNF uses {X} to mean "zero or more X". This maps directly and exclusively to a while loop, not recursion. Using recursion here works but creates unnecessary stack depth and obscures the structure.

// Grammar: param-list = param {',' param}
param_list():
  param()
  while tok == ','
    match(',')
    param()

// Grammar: stmts = {stmt}
stmts():
  while tok in FIRST(stmt)
    stmt()
The while condition is always "is the current token in FIRST of what we're repeating?" If FIRST(stmt) = {IF, WHILE, ID, ...}, then loop while tok is one of those. When tok is something outside that set, the loop exits.

Operator Precedence Through Grammar Structure

This is one of the most elegant things in compiler design. You get correct operator precedence for free, just by structuring the grammar correctly. No special precedence tables needed in the parser.

The rule: higher-precedence operators go deeper in the grammar hierarchy (further from the start symbol). Lower-precedence operators sit higher up.

expr   = term   {('+' | '-') term}    // + and - bind loosest
term   = factor {'*' factor}           // * binds tighter
factor = '(' expr ')' | id | num       // atoms at the bottom

Why does this give correct precedence? Trace parsing a + b * c:

expr()
  term()        // parses 'a' -- call stack goes: expr -> term -> factor
  // sees '+', enters while loop
  match('+')
  term()        // parses 'b * c' entirely inside term()
    factor()    // parses 'b'
    // sees '*', enters while loop
    match('*')
    factor()    // parses 'c'
    // loop ends: next token is not '*'
  // back in expr: 'b * c' was grouped as one unit
  // loop ends: no more '+' or '-'
Multiplication binds tighter because term() fully completes before expr() ever sees the next + or -. The call stack is your implicit parse tree. Deep calls group tightly.
Grammar Level Handles Precedence
expr (top) +, - Lowest
term (middle) *, / Medium
factor (bottom) atoms, parenthesized expressions Highest

Error Handling

Three layers of error detection in a recursive descent parser:

1. Failed match()

The simplest case. match() detects that the current token is not what the grammar demands at this point.

match(expected):
  if tok != expected
    error("line " + line + ": expected " + expected + ", got " + tok)

2. Default branches

Inside every alternation, the final else handles unrecognized tokens. This catches tokens that belong to no known production.

stmt():
  if tok == IF     ...
  else if tok == WHILE  ...
  else
    error("unexpected token: " + tok)  // catches everything else

3. Return codes

Functions return a status (1 = success, 0 = failure). The caller checks the return value and propagates errors up the call chain without crashing immediately.

oprnd():
  if reg() != 0        // reg() returned 0 means failure
    if tok == ','
      match(',')
      return num()
    return 1
  return 1             // no operand is also valid (epsilon production)
Panic mode recovery: on a serious error, skip tokens until you find a "synchronization token" (like ; or }) and restart parsing from there. This lets the parser report multiple errors in one pass rather than stopping at the first one. In exam questions, just know the concept: find a safe restart point and continue.

Complete Example: ASM Parser

The course uses an assembly language parser as a complete worked example. Here is the grammar and the direct translation:

// Grammar:
// asm    = op oprnd asm | EOF
// op     = CLRA | MOV | ADD | MOVA
// oprnd  = reg ',' num | reg | empty
// reg    = R[0..31]
reg():
  match('R')
  return num()       // returns the register number

op():
  switch tokentype()
    case MOV:  match('MOV');  return 1
    case ADD:  match('ADD');  return 1
    case CLRA: match('CLRA'); return 1
    case MOVA: match('MOVA'); return 1
    default:   return 0       // not a valid opcode

oprnd():
  if reg() != 0          // try to parse a register
    if tok == ','        // lookahead: is there a second operand?
      match(',')
      return num()       // reg , num production
    return 1             // just reg production
  return 1               // empty production (epsilon)

asm():
  if tok == EOF   return 1          // base case
  if op() != 0                      // try to parse an instruction
    oprnd()
    asm()                           // recurse for next instruction
  return 0
In oprnd(), the parser first tries to parse a register. If that succeeds, it checks for a comma using lookahead to decide between reg , num and reg alone. This single-token lookahead is exactly LL(1) in action. The comma is in the lookahead, not yet consumed.

Exam Checklist

  • Given a grammar, I can write all parser functions directly from the grammar rules
  • I know: sequence = sequential calls, alternation = if/else on tok, repetition = while loop
  • I can write the match() function from memory and explain what it does
  • I can trace parser execution step-by-step for a given input string
  • I understand that grammar depth = operator precedence (deeper = tighter binding)
  • I can handle lookahead decisions: peek at tok, then call match() to consume it
  • I can explain the three error handling techniques: failed match, default branch, return codes
