Introduction to Compilers

The big picture: what a compiler is, how its phases connect, and why the pipeline structure underlies everything else in this course.

Compiler vs. Interpreter

Both translate source code -- but they differ in when execution happens.

Compiler Interpreter
Strategy Translate the entire program first, then run Walk the parse tree and execute instructions live
When execution happens After translation is complete During the translation process
Output Machine code or intermediate code Direct results -- no persistent output
Examples GCC, Clang, Go compiler Python (CPython), Kamin's SLIC interpreter
A code generator is structurally an interpreter. The difference: instead of executing each node of the parse tree, it emits the machine instructions that will produce the same result later. Same tree traversal logic, different output action.

The 6-Phase Compilation Pipeline

Every compiler transforms source text through six sequential phases. Each phase takes one representation and produces another. Nothing skips ahead.

Source Characters → Scanner → Tokens → Parser → Parse Tree → Semantic Analyzer → Annotated Tree → IR Code Gen → Intermediate Code → Optimizer → Optimized Code → Code Generator → Target Machine Code
Phase Input Output What it does
1. Scanner Character stream Token stream Groups characters into meaningful lexical units (identifiers, numbers, operators). Discards whitespace and comments.
2. Parser Token stream Parse tree / AST Checks that the token sequence obeys the grammar. Builds a tree encoding the syntactic structure.
3. Semantic Analyzer Parse tree Annotated tree Checks meaning, primarily type correctness. Annotates each node with type information.
4. Intermediate Code Gen Annotated tree Three-address code (IR) Translates the tree into a flat, machine-independent instruction sequence. The "universal handoff" between front and back end.
5. Code Optimizer Intermediate code Optimized IR Replaces inefficient instruction patterns with equivalent but faster ones. Eliminates unnecessary temporaries, redundant loads, etc.
6. Code Generator Optimized IR Target machine code Maps IR instructions to actual CPU instructions for the target architecture. Handles register allocation and instruction selection.

Front-End vs. Back-End Split

The intermediate representation is a clean seam that divides the compiler into two independent halves.

Front-End Back-End
Phases included Scanner, Parser, Semantic Analyzer, IR Code Gen Code Optimizer, Code Generator
Depends on Source language rules and grammar Target hardware architecture
Reusability One front-end serves multiple back-ends (same language, different CPUs) One back-end serves multiple front-ends (different languages, same CPU)
Real-world example LLVM: Clang (C/C++), Rust, Swift all share the same LLVM back-end GCC: C, C++, Fortran, Ada all compile to the same optimizer + code gen
LLVM is the canonical proof that this split works industrially. Apple, Google, and Mozilla all build on it. The intermediate representation (LLVM IR) is the contract -- the front-end produces it, the back-end consumes it. Neither side needs to know anything about the other.

Concrete Data Flow Example: x = y + 3

Trace one assignment statement through all six phases to see exactly what changes at each step.

-- Source input --
x = y + 3


-- Phase 1: Scanner --
Tokens: [id:x]  [=]  [id:y]  [+]  [num:3]

Groups raw characters into labeled units.
Whitespace is gone. "x" becomes token id:x, "3" becomes num:3.


-- Phase 2: Parser --
Parse tree:
    (= x (+ y 3))

Tokens arranged into a tree that encodes operator precedence and structure.
"+" binds y and 3 before "=" assigns to x.


-- Phase 3: Semantic Analyzer --
Annotated tree:
    (= x:int (+ y:int 3:int))

Type annotations added to every node.
Type checker confirms: int + int = int, assignable to int x. No errors.


-- Phase 4: Intermediate Code Generation --
t1 = y + 3
x  = t1

Tree flattened into three-address instructions.
Each instruction has at most one operator and three operands.
Temporary variable t1 introduced to hold the intermediate result.


-- Phase 5: Code Optimizer --
x = y + 3

Temporary t1 eliminated -- "copy propagation" optimization.
Semantically identical, but one fewer instruction.


-- Phase 6: Code Generator --
ld.y       ; load y into accumulator
lit.3      ; push literal 3
add        ; add top two stack values
st.x       ; store result into x

Final machine instructions for target CPU.
Stack-based machine model used here (like Kamin's VM).
The pipeline is a chain of representation changes. Source text becomes tokens, tokens become a tree, the tree gains types, the typed tree becomes flat IR, IR gets cleaned up, IR becomes machine code. Every exam topic maps to exactly one phase: scanning = phase 1, CFG/parsing = phase 2, FIRST/FOLLOW = phase 2 mechanics, type checking = phase 3, activation records = runtime support for phase 6 output. Know the pipeline and you know where everything lives.
Students confuse "parse tree" and "AST." A parse tree includes every grammar symbol including parentheses and semicolons. An Abstract Syntax Tree (AST) strips the grammar scaffolding and keeps only the semantic structure. Most real compilers build an AST, not a raw parse tree. They serve the same purpose -- know the distinction when asked.

Why the Pipeline Structure Matters for This Course

Every major topic in this course is one phase of this pipeline examined in depth:

  • CFG and Left Recursion -- the grammar formalism the Parser operates on (Phase 2)
  • FIRST and FOLLOW sets -- the lookahead mechanism that drives Parser decisions (Phase 2)
  • Recursive Descent Parsing -- one concrete algorithm for implementing Phase 2
  • Code Generation and Three-Address Code -- how Phases 4 and 6 work
  • Activation Records -- the runtime memory layout that makes generated code correct (support for Phase 6)

Understanding the pipeline gives you a cognitive map. When you encounter a new concept, you can immediately ask: which phase does this belong to? What is its input? What does it produce? That question alone will orient you on every exam problem.
