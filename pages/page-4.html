FIRST and FOLLOW Sets

The lookup tables that make top-down parsing deterministic

Master two definitions and three rules each. Everything else in LL(1) parsing flows from them. If you can compute FIRST and FOLLOW from scratch, you can build any LL(1) parse table.

Why These Sets Exist

A top-down parser reads one token and must immediately decide which production to expand. Without a systematic way to answer "which production leads to something starting with this token?", the parser would guess or backtrack. FIRST and FOLLOW eliminate that uncertainty.

FIRST(A) answers: "what tokens can A start with?" FOLLOW(A) answers: "what tokens can come right after A?" Together they let the parser make a deterministic choice at every step.
Grammar → Compute FIRST → Compute FOLLOW → Build Parse Table M[A, t] → LL(1) Parser

FIRST Sets

FIRST(X) is the set of all terminals that can appear as the first symbol of any string derived from X. If X can derive the empty string (epsilon), then epsilon is also in FIRST(X).

Computation Rules

CaseRule
X is a terminal FIRST(X) = { X }
X is epsilon FIRST(epsilon) = { epsilon }
X is a nonterminal with production X → Y1 Y2 ... Yn Add FIRST(Y1) minus epsilon. If epsilon in FIRST(Y1), also add FIRST(Y2) minus epsilon. Continue until either epsilon is not in FIRST(Yi) or you exhaust all Yi. Add epsilon only if epsilon is in FIRST(Yi) for all i.

The sequence rule is the critical one. Think of it as: "keep scanning right as long as the current symbol can vanish (derive epsilon)."

FIRST(Y1 Y2 ... Yn):
  result = {}
  for each Yi:
    result = result + (FIRST(Yi) - {epsilon})
    if epsilon not in FIRST(Yi):
      return result          // Yi cannot vanish; stop here
  result = result + {epsilon}  // all symbols can vanish
  return result
Epsilon propagates rightward, not leftward. If Y1 can derive epsilon, Y2 contributes to FIRST. But FIRST(Y1 Y2) contains epsilon only if BOTH Y1 and Y2 can derive epsilon. Many students forget the final epsilon addition step.

FOLLOW Sets

FOLLOW(A) is the set of terminals (including $, the end-of-input marker) that can appear immediately to the right of A in some sentential form. Epsilon is never in a FOLLOW set.

Computation Rules

RuleConditionAction
Rule 1 S is the start symbol Add $ to FOLLOW(S)
Rule 2 Production B → ...A t... where t is a terminal Add t to FOLLOW(A)
Rule 2b Production B → ...A C... where C is a nonterminal Add FIRST(C) minus epsilon to FOLLOW(A). If epsilon in FIRST(C), also add FOLLOW(B) to FOLLOW(A).
Rule 3 Production B → ...A (A is at the end, or everything after A can derive epsilon) Add FOLLOW(B) to FOLLOW(A)
Rule 3 is where FOLLOW sets talk to each other. "If A ends a production for B, then whatever can follow B can also follow A." This creates dependencies between FOLLOW sets and is why we need fixed-point iteration.

The Fixed-Point Algorithm

Both FIRST and FOLLOW are computed by the same iterative strategy: start with empty sets, apply rules, repeat until nothing changes.

// Generic fixed-point iteration
for each nonterminal A:
  FIRST(A) = {}    // or FOLLOW(A) = {}

changed = true
while changed:
  changed = false
  for each production A -> alpha:
    new_tokens = apply_rules(A, alpha)
    if new_tokens is not a subset of FIRST(A):
      FIRST(A) = FIRST(A) + new_tokens
      changed = true

This terminates because:

  • Sets only grow (tokens are only added, never removed)
  • The maximum size of any set is bounded by the number of terminals
  • Therefore, after at most |terminals| + 1 rounds, nothing can change
Fixed-point iteration is a general technique in static analysis. You apply rules until a stable state (the "fixed point") is reached. The same pattern appears in dataflow analysis, type inference, and constraint solving.

Worked Example: Computing FIRST and FOLLOW

Grammar

Expr    -> Expr + Term | Term
Term    -> Term * Primary | Primary
Primary -> FunCall | Id | ( Expr )
FunCall -> Id ( OptArgs )
OptArgs -> epsilon | Args
Args    -> Expr | Args , Expr

FIRST Sets: Iteration by Iteration

Round 0: All FIRST sets are empty.

Round 1: Apply rules to terminals and base productions.

FIRST(Primary) = FIRST(FunCall) u FIRST(Id) u FIRST( ( Expr ) )
               = {} u {Id} u {(}
               = {Id, (}          // FunCall still empty this round

FIRST(FunCall) = {Id}             // starts with terminal Id

FIRST(OptArgs) = {epsilon}        // has epsilon production; Args still empty

FIRST(Args)    = {}               // FIRST(Expr) still empty

Round 2: Propagate through the chain.

FIRST(Term)    = FIRST(Primary) = {Id, (}
FIRST(Expr)    = FIRST(Term)    = {Id, (}
FIRST(FunCall) = {Id}             // unchanged

Round 3:

FIRST(Args)    = FIRST(Expr)    = {Id, (}
FIRST(OptArgs) = {epsilon, Id, (}  // epsilon + FIRST(Args)

Round 4: No changes. Fixed point reached.

Final FIRST Sets

NonterminalFIRST
Expr{ Id, ( }
Term{ Id, ( }
Primary{ Id, ( }
FunCall{ Id }
OptArgs{ epsilon, Id, ( }
Args{ Id, ( }

FOLLOW Sets: Iteration by Iteration

Round 0: FOLLOW(Expr) = { $ }, all others empty (Rule 1: Expr is start symbol).

Round 1: Scan every production for each rule.

From Expr -> Expr + Term:
  FOLLOW(Expr) += {+}          // Rule 2: + follows Expr
  FOLLOW(Term) += FOLLOW(Expr) // Rule 3: Term at end of production
               = {$, +}

From Primary -> ( Expr ):
  FOLLOW(Expr) += {)}          // Rule 2: ) follows Expr

From Term -> Term * Primary:
  FOLLOW(Term)    += {*}       // Rule 2: * follows Term
  FOLLOW(Primary) += FOLLOW(Term) // Rule 3: Primary at end

From FunCall -> Id ( OptArgs ):
  FOLLOW(OptArgs) += {)}       // Rule 2: ) follows OptArgs

From Args -> Args , Expr:
  FOLLOW(Args) += {,}          // Rule 2: , follows Args
  FOLLOW(Expr) += FOLLOW(Args) // but FOLLOW(Args) = {,} so far

After Round 1:

FOLLOW(Expr)    = {$, +, )}
FOLLOW(Term)    = {$, +, *}
FOLLOW(Primary) = {$, +, *}
FOLLOW(OptArgs) = {)}
FOLLOW(Args)    = {,}

Round 2: Re-apply with updated sets.

OptArgs -> Args: FOLLOW(Args) += FOLLOW(OptArgs) = {), ,}
Args -> Args , Expr: FOLLOW(Expr) += FOLLOW(Args) = {$, +, ), ,}
Expr -> ... Term: FOLLOW(Term) += FOLLOW(Expr) = {$, +, ), *}
Term -> ... Primary: FOLLOW(Primary) += FOLLOW(Term) = {$, +, ), *}

Round 3: Check FunCall.

Primary -> FunCall: FOLLOW(FunCall) += FOLLOW(Primary) = {$, +, ), *}

Round 4: No changes.

Final FOLLOW Sets

NonterminalFOLLOW
Expr{ $, +, ), , }
Term{ $, +, ), * }
Primary{ $, +, ), * }
FunCall{ $, +, ), * }
OptArgs{ ) }
Args{ ), , }

Building the LL(1) Parse Table

The table M[A, t] tells the parser: "when the top of the stack is nonterminal A and the next input token is t, use this production."

Table Construction Algorithm

for each production A -> alpha:
  for each terminal t in FIRST(alpha) - {epsilon}:
    M[A, t] = "A -> alpha"
  if epsilon in FIRST(alpha):
    for each terminal t in FOLLOW(A):  // including $
      M[A, t] = "A -> alpha"

Intuition: if production A → alpha can start with t, use it when you see t. If alpha can vanish (derives epsilon), use it when you see something that can follow A -- meaning A will disappear and let that token be matched later.

Partial Parse Table for OptArgs

NonterminalId()$+,
OptArgs OptArgs → Args OptArgs → Args OptArgs → epsilon
Args Args → Expr Args → Expr
The epsilon production for OptArgs goes into the ) column -- NOT because ) is in FIRST(epsilon), but because ) is in FOLLOW(OptArgs). This is Rule 2 of table construction: epsilon productions use FOLLOW.

LL(1) Conflicts

A grammar is LL(1) if and only if every cell M[A, t] has at most one production. A conflict means two productions compete for the same (nonterminal, token) pair -- the parser cannot decide deterministically.

Two Root Causes

CauseExampleSymptom
Left recursion A → A x | y Both productions contribute the same tokens to M[A, t]
Common prefix A → x B | x C Both productions have x in FIRST, so M[A, x] gets two entries

Detecting Conflicts

When filling the table, if you try to add a production to a cell that already has one: that is a conflict. The grammar is not LL(1) as written.

Left recursion causes conflicts because A → A x and A → y have overlapping FIRST sets: both can lead to y's FIRST tokens. The fix (left recursion elimination) was covered in the previous section. The remaining conflict type -- common prefix -- is fixed by left factoring.

Left Factoring to Resolve Conflicts

Left factoring pulls out the shared prefix into a separate production, deferring the choice to a new nonterminal.

The Pattern

// Before: common prefix x causes conflict
A -> x B
A -> x C

// After: left factored
A  -> x A'
A' -> B
A' -> C

Classic Example: Dangling Else

// Before: both start with "if ( E ) Stmt"
Stmt -> if ( E ) Stmt
Stmt -> if ( E ) Stmt else Stmt

// After: left factor on "if ( E ) Stmt"
Stmt  -> if ( E ) Stmt Stmt'
Stmt' -> else Stmt
Stmt' -> epsilon

Now M[Stmt, if] has exactly one production. M[Stmt', else] = "Stmt' → else Stmt" and M[Stmt', $] = M[Stmt', ...] = "Stmt' → epsilon" for anything in FOLLOW(Stmt').

Left factoring does not always work. If after left factoring you still have conflicts, the grammar may be inherently ambiguous or may require more than one token of lookahead (LL(k) for k > 1, or a different parsing strategy entirely).

General Left Factoring Procedure

1. Find all productions for A that share a common prefix alpha:
   A -> alpha beta_1 | alpha beta_2 | ... | gamma (no shared prefix)

2. Replace with:
   A  -> alpha A' | gamma
   A' -> beta_1 | beta_2 | ...

3. Repeat until no two productions for any nonterminal share a prefix.

Exam Readiness Checklist

  • Compute FIRST of a sequence X1 X2 ... Xn correctly, propagating epsilon rightward
  • Apply all three FOLLOW rules: start symbol gets $, look right for terminals, propagate from enclosing production at end
  • Run fixed-point iteration: start empty, apply rules, stop when nothing changes
  • Build an LL(1) table: epsilon productions use FOLLOW, others use FIRST
  • Identify a conflict: two productions in one cell means not LL(1)
  • Apply left factoring to resolve a common-prefix conflict
