Scanner (Lexical Analysis)

The first phase: turning a raw character stream into a structured token stream

The scanner is conceptually the simplest compiler phase, yet it teaches the fundamental abstraction the entire field rests on: regular languages. Master tokens, patterns, lexemes, and the regex → NFA → DFA pipeline, and lexical analysis is done.

What the Scanner Does

The scanner (lexer) reads source code one character at a time and groups characters into tokens -- the atomic vocabulary that the parser works with. It strips whitespace and comments, so the parser never sees them.

Source characters → Scanner → Token stream → Parser

Concrete example -- one line of source, nine tokens out:

Input:   if (age >= 18) grade = 3.14

Output:  [if]  [(]  [id:age]  [>=]  [num:18]  [)]  [id:grade]  [=]  [num:3.14]
The parser never touches raw characters. Tokens are the agreed-upon interface between the scanner and every phase that follows. Design this interface well, and the rest of the compiler stays clean.

The Trinity: Token, Pattern, Lexeme

Three terms that beginners conflate. They are completely different things.

Term What it is Analogy Example
Token A named category (a type) The word class "noun" identifier, number, if
Pattern The rule (regex) that defines valid strings for a token The grammar rule for nouns letter (letter | digit)*
Lexeme The actual string from source code that matched The specific word "dog" age, grade, q1

Read it as: the token identifier has the pattern letter(letter|digit)*, and age, grade, q1 are all lexemes for that token.

Common token/lexeme pairs from lecture

TokenPatternExample Lexemes
identifierletter (letter | digit)*Age, grade, q1
number(+|-)? digit+ (. digit+)? (E (+|-)? digit+)?3.1416, -498127, 1E10
keyword ifexact string ifif, IF (if case-insensitive)
>=exact string >=>=
Token and lexeme are not synonyms. identifier is a token (a category). age is a lexeme (a specific string). Mixing them up is a guaranteed point loss on exams.

Regular Expressions for Token Patterns

Regular expressions are the formal tool for writing patterns. The core operators:

OperatorMeaningExample
abConcatenation: a then bif
a | bAlternation: a or bletter | digit
a*Kleene star: zero or more adigit*
a+One or more a (shorthand for aa*)digit+
a?Zero or one a (optional)(+|-)?
(a)Grouping(+|-)?

The full number pattern, broken down:

number  =  (+ | -)?  digit+  (. digit+)?  (E (+ | -)? digit+)?
           ^sign     ^integer ^fraction    ^scientific notation
Regular expressions define exactly the class of languages recognizable by finite automata. That equivalence is why RE → NFA → DFA works at all. It is not a trick -- it is a theorem.

The Regex → NFA → DFA Pipeline

Writing a regex is the human-friendly part. Running it efficiently requires converting it into a machine. The pipeline:

Regex (per token) → NFA (Thompson's) → Combined NFA → DFA (subset construction) → Scanner code

Step by step

StepWhat happensKey property
1. Write regex One regex per token type Human-readable specification
2. Build NFA per regex Thompson's construction: mechanical, one fragment per operator Nondeterministic -- multiple active states at once
3. Combine NFAs New start state with epsilon transitions to each token's NFA One machine handles all tokens simultaneously
4. NFA → DFA Subset construction: each DFA state = a set of NFA states Deterministic -- exactly one active state, O(n) scan
5. Implement DFA Either nested switch/if or a 2D transition table Ready to run on real input

Two implementation strategies for the DFA

StrategyHow it worksTrade-off
Nested switch / if State machine encoded directly in control flow; each state is a case Fast, easy to read; hard to regenerate automatically
Transition table next_state = table[current_state][input_char] Easy to auto-generate (lex/flex); one extra memory lookup per character 
/* Transition table lookup -- the core loop */
state = START;
while ((c = next_char()) != EOF) {
    state = table[state][c];
    if (is_accept[state]) emit_token(state);
}
You do not need to memorize Thompson's construction step-by-step for most exams. What you must know: (1) regex defines the language, (2) NFA is the intermediate built from regex, (3) DFA runs it deterministically, (4) direction of transformation is always regex → NFA → DFA, never backward.

Disambiguating Rules

Real input is ambiguous. Two rules resolve every conflict:

Rule 1: Longest Match (Maximal Munch)

Always consume the longest possible string that matches any token pattern. The scanner reads ahead until no longer match is possible, then returns that longest match.

Input: <=
  Naive:   token '<'  then  token '='    -- WRONG
  Correct: token '<='  (one token)        -- RIGHT

Input: ifx
  Naive:   keyword 'if'  then  id 'x'   -- WRONG
  Correct: identifier 'ifx'              -- RIGHT

Rule 2: Keyword Priority

When the longest match ties between a keyword and an identifier, the keyword wins. if matches the identifier pattern letter(letter|digit)* -- but it is recognized as the keyword if, not as an identifier.

In practice: scan first, then check a reserved-word table. If the lexeme is in the table, emit the keyword token; otherwise emit identifier.

Longest match is applied before keyword priority. First find the longest match, then check if it is a keyword. Getting the order wrong produces wrong tokens.

Scanner-Parser Interaction

The scanner is not autonomous. It is called on demand by the parser:

Parser needs token → Calls scanner → Scanner reads chars → Returns (token, lexeme, attr) → Parser decides

The scanner may also evaluate attributes -- for example, converting the string "3.14" to the float value 3.14, or recording the variable name age into the symbol table. These attributes travel with the token to later phases.

           |     |      |       |    |    |    |

           |     |      |       |        |    |
        keyword punc  ident   relop    num   punc
        token   token token   token   token  token

Notice that >= is one token, not two. Longest match at work. And if is a keyword token, not an identifier, even though it satisfies the identifier pattern. Both disambiguation rules applied in this one short example.

Exam Checklist

  • Distinguish token (category), pattern (regex rule), and lexeme (actual string)
  • Write a regex for identifier: letter (letter | digit)*
  • Write a regex for number including optional sign, fraction, and exponent
  • State the regex → NFA → DFA direction (never backward)
  • Apply longest match: <= is one token, ifx is an identifier
  • Apply keyword priority: if matching identifier pattern still becomes keyword token
  • Explain that the parser calls the scanner on demand (scanner is passive)
