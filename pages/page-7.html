Code Generation & S-Code

The compiler is an interpreter that writes instructions instead of running them.

Master these five patterns and you can generate S-code for any program on the exam. Everything else is variation on these themes: arithmetic, globals, if-else with patching, function calls, and recursion.

The Pipeline

Source Code → Parse Tree → Code Generator → S-Code (bytecode) → S-Machine executes

S-Code Instruction Format

Every S-code instruction is a fixed-width 32-bit word, split into two fields:

[ argument: 24 bits | opcode: 8 bits ]
  ^-- numeric arg    ^-- what to do
8 bits for opcode = 256 possible instructions. 24 bits for argument = values up to 16 million. A single word encodes both operation and operand. No variable-length encoding, no register fields -- just opcode + one inline argument.

S-code is a zero-address (stack) machine. There are no register operands in instructions. All computation goes through the evaluation stack. Two runtime data structures matter:

  • Evaluation stack (global): operands are pushed, operations pop and push results
  • Activation record (per call): local variables, return address, saved AR pointer

S-Code Instruction Set

Data Movement

Instruction Action Use When
get.N Push local variable #N onto stack Reading a local variable
put.N Pop stack into local variable #N Assigning to a local variable
ld.x Push global variable x onto stack Reading a global variable
st.x Pop stack into global variable x Assigning to a global variable
lit.N Push literal constant N onto stack Using a numeric constant
ldx Load indexed (array read) Array access
stx Store indexed (array write) Array assignment

Arithmetic, Logic, and Comparison

Category Instructions
Arithmetic add, sub, mul, div, mod
Bitwise / Logic band, bor, bxor, shl, shr, not
Comparison eq, ne, lt, le, gt, ge
Increment / Decrement inc, dec

All binary operations follow the same contract: pop 2, push 1. The left operand must be pushed first, the right operand second.

Control Flow

Instruction Action
jf.L Pop stack; if false (0), jump to label L
jt.L Pop stack; if true (non-zero), jump to label L
jmp.L Unconditional jump to label L

Function Support

Instruction Action
fun.x Function header -- allocate x local variable slots
call.f Call function f (arguments already pushed onto stack)
ret.y Return from function (y encodes cleanup: arg count + 1)
sys.N System call (e.g., sys.1 = print top of stack)

eval() vs codegen(): The Core Duality

An interpreter and a compiler walk the same parse tree with the same structure. The interpreter executes each node. The compiler emits instructions for each node. Same recursion, different action at the leaf.
Tree Node Interpreter eval() Compiler codegen()
Literal 5 Return the value 5 Emit lit.5
Variable a Look up a in environment, return value Emit get.N or ld.a
a + b eval(a) + eval(b), return sum codegen(a); codegen(b); emit add
if (c) t else e if eval(c) then eval(t) else eval(e) codegen(c); emit jf; codegen(t); emit jmp; codegen(e)

Here is the core code generation loop in pseudocode:

eval(e):
  if e == nil: stop
  if isatom(e): do_atom(e)          -- emit lit or get/ld

  a = head(e)                        -- operator
  b = tail(e)                        -- operands

  switch typeof(a):
    PLUS:
      eval(arg1(b))                  -- generate code for left
      eval(arg2(b))                  -- generate code for right
      out(ADD)                       -- emit the add instruction

    IF:
      eval(arg1(b))                  -- generate code for condition
      out(JF, 0)                     -- emit jf with placeholder target
      patch_addr = CP - 1            -- remember where to patch
      eval(arg2(b))                  -- generate code for then-branch
      patch(patch_addr, CP - patch_addr)  -- fill in correct offset

The Patching Technique for Forward Jumps

When emitting a conditional jump (jf), the compiler does not yet know where to jump -- the target is code that has not been generated yet. The solution is a two-pass approach done inline:

Emit jf.0 (placeholder) → Save address (CP-1) → Generate body code → Patch saved address with real target (CP)
Forgetting to emit the jmp at the end of the then-branch is the most common exam mistake. Without it, control falls through into the else-branch even when the condition was true. Every if-else needs two jumps: one jf and one jmp, each patched separately.

The Five Code Generation Patterns

Pattern 1: Arithmetic with Local Variables

Local variables are accessed by slot number within the current activation record using get (read) and put (write).

Source: a = b + c with locals a=#1, b=#2, c=#3

get.2       -- push b onto stack
get.3       -- push c onto stack
add         -- pop b and c, push (b + c)
put.1       -- pop result, store into a

Stack trace:

After get.2:  [ b ]
After get.3:  [ b | c ]
After add:    [ b+c ]
After put.1:  [ ]        -- a now holds b+c

Pattern 2: Arithmetic with Global Variables

Global variables are accessed by name using ld (read) and st (write). The pattern is identical to locals -- only the instruction names differ.

Source: a = b + c with globals a, b, c

ld.b        -- push global b
ld.c        -- push global c
add         -- pop b and c, push (b + c)
st.a        -- pop result, store into global a
The pattern is symmetric: read both operands, apply the operator, write the result. Whether local or global changes only the load/store instruction, not the structure.

Pattern 3: If-Else with Patching

Source: if (a == b) c = 1; else c = 2; with locals a=#1, b=#2, c=#3

get.1       -- push a
get.2       -- push b
eq          -- pop a and b, push 1 if equal else 0
jf.xx       -- if false (a != b), jump to else-branch
lit.1       -- then-branch: push 1
put.3       -- c = 1
jmp.yy      -- skip over else-branch
:xx         -- else label (patched target of jf)
lit.2       -- else-branch: push 2
put.3       -- c = 2
:yy         -- end label (patched target of jmp)

During code generation, xx and yy are initially 0. After generating each branch, the compiler patches them with the actual code pointer values.

Pattern 4: Function Definition and Call

Arguments are pushed onto the stack before the call. The callee accesses them as local variables starting at slot #1.

Source:

mysum(a, b) { return a + b; }
main() { mysum(3, 4); }

S-code:

:mysum
    fun.x       -- allocate local variable slots (x = frame size)
    get.1       -- push parameter a (slot 1)
    get.2       -- push parameter b (slot 2)
    add         -- a + b
    ret.y       -- return (result stays on stack for caller)

:main
    fun.x       -- main's frame
    lit.3       -- push first argument
    lit.4       -- push second argument
    call.mysum  -- call (args already on stack before call)
    ret.y       -- return from main
Arguments are pushed in order (first arg pushed first) BEFORE the call. Inside the function, the first argument is local slot #1. Do not emit call before pushing all arguments.

Pattern 5: Recursion

Recursion requires no special mechanism -- the call instruction works the same for self-calls. A new activation record is created for each call. The key challenge is tracking which local slot is which parameter.

Source:

sum(n, m) {
    if (n == m) return m;
    else return n + sum(n+1, m);
}
main() { print(sum(1, 10)); }

S-code (parameter mapping: n=#2, m=#1 -- check your compiler's convention):

:main
    fun.1
    lit.1
    lit.10
    call.sum
    sys.1       -- print result
    ret.1

:sum
    fun.1
    get.2       -- push n
    get.1       -- push m
    eq          -- n == m?
    jf.L18      -- if false, jump to else
    get.1       -- base case: push m
    ret.3       -- return m
    jmp.L26     -- (dead code after return, but emit for structure)
:L18
    get.2       -- push n  (for the outer addition)
    get.2       -- push n  (for the n+1 argument to recursive call)
    lit.1
    add         -- n + 1
    get.1       -- push m
    call.sum    -- sum(n+1, m)
    add         -- n + sum(n+1, m)
    ret.3       -- return result
:L26
    ret.3
Notice that n is pushed twice before the recursive call: once to hold its value for the outer add, and once to compute n+1. The stack is your only scratchpad. If you need a value after a call, push a copy before the call.

Pattern Summary

Pattern Key Instructions Critical Rule
Arithmetic (local) get.N, put.N, op Push both operands before the op
Arithmetic (global) ld.x, st.x, op Same structure, different load/store
If-else jf, jmp, patch Two jumps, two patches; don't forget jmp
Function call lit/get, call.f, ret Push all args before call
Recursion Same as call, plus jf for base case Push extra copies if value needed after recursive call

Exam Checklist

  • I can translate a = b + c to S-code for both local (get/put) and global (ld/st) variables.
  • I can generate S-code for if-then-else with correct jf, jmp, and two patched labels.
  • I can generate S-code for a function call (push args first, then call.f).
  • I can generate S-code for a recursive function, handling the base case and the self-call separately.
  • I understand the patching technique: emit placeholder, save address, generate body, overwrite placeholder with real target.
  • I can trace the evaluation stack state step-by-step through a sequence of S-code instructions.
  • I understand that codegen() and eval() walk the same tree -- they differ only in what they do at each node.
