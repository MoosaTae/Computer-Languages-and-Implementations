The 6 Pareto Concepts

1. The Compilation Pipeline

The 6 phases: Scanner → Parser → Semantic → IR Gen → Optimizer → Code Gen. Every other topic is a deep dive into one of these phases. Know the pipeline = know where everything fits.

2. FIRST/FOLLOW + LL(1) Parse Table

Computing FIRST/FOLLOW sets using iterative fixed-point algorithm, then building the parse table. This is the mathematical backbone of all top-down parsing.

3. Grammar → Parser → Code

Left-recursion removal, then one-function-per-nonterminal translation. The grammar IS the code. This is the core skill -- tested heavily.

4. S-code Generation Patterns

get/put for locals, ld/st for globals, lit for constants, jf/jmp for control flow, patching for forward jumps. Know the 5 patterns (arithmetic local/global, if-else, function call, recursion).

AR fields, pre-call/prolog/epilog/post-return, access link vs control link, static coordinates. One diagram drawn correctly answers a dozen sub-questions.

6. Two-Stack Parse Tree Construction

Argument stack + operator stack, makebintree() function, action routines in parser. Bridges parsing and code generation.

Introduction to Compilers

What a compiler actually is, and why the pipeline structure is everything

Compiler vs. Interpreter

A compiler translates your entire source program into a target program. Run the target later, separately. The compiler is not present at runtime.

An interpreter walks the parse tree and executes it directly. No separate target. The interpreter is present at runtime, doing work every execution.

A code generator works exactly like an interpreter, except instead of executing each operation, it outputs an instruction for it. Same tree walk. Different output destination. This insight connects parsing to code generation.
CompilerInterpreter
What it doesTranslates entire source to target programExecutes parse tree directly
When execution happensAfter translation is completeDuring the translation
OutputMachine code / intermediate codeDirect results

The 6 Compilation Phases

Characters → Scanner → Tokens → Parser → Parse Tree → Semantic → Annotated Tree → IR Gen → Optimizer → Code Gen → Target Code
PhaseInputOutputPurpose
ScannerCharactersTokensGroup chars into meaningful tokens
ParserTokensParse treeCheck syntax against grammar rules
Semantic AnalyzerParse treeAnnotated treeType checking, scope resolution
IR GeneratorAnnotated treeThree-address codeMachine-independent representation
OptimizerIR codeOptimized IRReplace inefficient sequences
Code GeneratorOptimized IRTarget codeProduce final machine instructions

Front-End vs. Back-End

FRONT-END (Language-specific) BACK-END (Hardware-specific) +----------------------------+ +----------------------------+ | Scanner | | Code Optimizer | | Parser | -> | Code Generator | | Semantic Analyzer | | | | Intermediate Code Gen | | | +----------------------------+ +----------------------------+ Reusable for same language Different per target machine

One front-end serves multiple back-ends. One back-end serves multiple front-ends. This is how GCC/LLVM work.

Data Flow Example

Source:    "x = y + 3"
Scanner:   [id:x] [=] [id:y] [+] [num:3]
Parser:    (= x (+ y 3))
Semantic:  (= x:int (+ y:int 3:int))
IR Code:   t1 = y + 3; x = t1
Optimized: x = y + 3
Code Gen:  ld.y  lit.3  add  st.x
The pipeline is a chain of transformations. Each phase takes one representation and produces another. Understanding this chain means you understand where every other topic fits: CFG/Parsing = Parser phase, FIRST/FOLLOW = Parser decisions, Code Generation = back-end, Activation Records = runtime support.

  • Explain why code generator resembles an interpreter

Scanner (Lexical Analysis)

The first phase: turning a stream of characters into tokens

What the Scanner Does

Input:  "if (age >= 18) grade = 3.14"
Output: [if] [(] [id:age] [>=] [num:18] [)] [id:grade] [=] [num:3.14]

Tokens, Patterns, and Lexemes

TermDefinitionExample
TokenThe categoryidentifier, number, keyword
PatternThe regex ruleletter(letter|digit)*
LexemeThe actual string matchedage, grade, 3.14

Regex Patterns

Identifier:  letter ( letter | digit )*
Number:      (+|-)? digit+ (. digit+)? (E (+|-)? digit+)?
Keyword:     if | else | while | for | return

Construction Pipeline

Regex → NFA (per token) → Combined NFA → DFA → Scanner Code
ImplementationHow it works
Nested-if / switchSwitch on current state and character
Transition table2D matrix: [state][char] → next_state

Disambiguating Rules

Longest Match (Maximal Munch): Always consume the longest possible token. <= is one token, not two. ifstream is an identifier, not keyword if + stream.

Keyword Priority: If longest match ties between keyword and identifier, keyword wins.

Scanner-Parser Interaction

Parser: "I need the next token" | v Scanner: reads characters --> returns token + attributes | v Parser: uses token to make parsing decisions Scanner is LAZY (waits). Parser is ACTIVE (requests).
3 things that matter: (1) Tokens are the interface between scanner and parser. (2) Regex → NFA → DFA is the pipeline. (3) Longest match resolves ambiguity.

  • Distinguish token, pattern, and lexeme with examples
  • Write regex for identifiers and numbers
  • Trace regex → NFA → DFA pipeline
  • Explain longest substring principle
  • Explain scanner-parser interaction

Context-Free Grammars & Left Recursion

The grammar IS the language. Get this wrong and your parser loops forever.

CFG Anatomy

  • Terminal -- actual tokens: id, +, (
  • Non-terminal -- abstract category: Expr, Term
  • Start symbol -- top-level non-terminal
  • Production -- rewrite rule: A → ...

Left Recursion: The Infinite Loop Trap

A rule like expr → expr op term | term means parseExpr() immediately calls parseExpr() before consuming any input. Stack overflow. Every time.

The Fix: Before & After

BEFORE (left-recursive):          AFTER (EBNF, safe):
  ex = ex + term | term             ex = term { (+ | -) term }
  term = term * factor | factor     term = factor { * factor }
  factor = ( ex ) | id              factor = ( ex ) | id

General pattern:
  A = A x | y  -->  A = y { x }

BNF vs EBNF

NotationMeaningCode equivalent
::=produces (same as →)function definition
|alternation (or)if/else
{ }zero or more (EBNF)while loop
[ ]optional (EBNF)if statement

Ambiguous Grammars

A grammar is ambiguous if one string has two valid parse trees.

Ambiguous:  E -> E + E | E * E | id
String "id + id * id" has TWO trees -- one groups + first, one groups * first.

Fix: use grammar hierarchy for precedence.
  expr = term { + term }      // low precedence at top
  term = factor { * factor }  // high precedence deeper
  factor = id | ( expr )
ProblemFix
Operator precedenceHigher-precedence ops deeper in grammar
Left associativityLeft-recursive rules (or EBNF {})
Dangling elseSplit into MatchedSt vs UnmatchedSt

Derivations

TypeRuleUsed by
LeftmostReplace leftmost non-terminal firstTop-down parsers
RightmostReplace rightmost non-terminal firstBottom-up parsers (in reverse)

Indirect Left Recursion

A => X =>* A Y -- A derives itself through other non-terminals. Good news from class: "never seen in real programming languages." Direct removal is sufficient.

3 essential skills: (1) Identify left recursion. (2) Remove it: A = Ax | y becomes A = y{x}. (3) Understand and resolve ambiguity via grammar hierarchy.

  • Convert between BNF and EBNF notation
  • Identify ambiguous grammar by showing two parse trees
  • Resolve ambiguity with precedence and associativity
  • Explain dangling else and the fix

FIRST and FOLLOW Sets

The two sets that make top-down parsing mechanical. Master these, and LL(1) tables write themselves.

FIRST tells you which production to choose. FOLLOW tells you when to use the epsilon production.

FIRST Set Rules

CaseRule
Terminal tFIRST(t) = {t}
EpsilonFIRST(ε) = {ε}
A → α1 | α2FIRST(A) = FIRST(α1) ∪ FIRST(α2)
Sequence X1X2...XnStart with FIRST(X1). If ε in FIRST(X1), add FIRST(X2), etc. ε in result only if in ALL.

FOLLOW Set Rules

#ConditionAction
1A is start symbolAdd $ to FOLLOW(A)
2B → ...A t ...Add t to FOLLOW(A)
3B → ...A C and ε in FIRST(C)Add FOLLOW(B) to FOLLOW(A)
4B → ...A (A at end)Add FOLLOW(B) to FOLLOW(A)

Algorithm: Fixed-Point Iteration

1. Initialize all FIRST/FOLLOW sets to empty
2. Apply all rules exhaustively -- add every token you can
3. Repeat until NO set changes
Terminates because sets only grow and token universe is finite.

Worked Example

Grammar:
  exp    -> term exp'
  exp'   -> addop term exp' | lambda
  addop  -> + | -
  term   -> factor term'
  term'  -> mulop factor term' | lambda
  mulop  -> *
  factor -> ( exp ) | num

Final FIRST Sets

Non-terminalFIRST
exp, term, factor{ (, num }
exp'{ +, -, ε }
term'{ *, ε }
addop{ +, - }
mulop{ * }

Final FOLLOW Sets

Non-terminalFOLLOW
exp, exp'{ $, ) }
term, term'{ +, -, $, ) }
factor{ *, +, -, $, ) }
addop, mulop{ (, num }

Building the LL(1) Parse Table

For each production A → α:

  • For each terminal a in FIRST(α): add A → α to M[A, a]
  • If ε in FIRST(α): for each terminal b in FOLLOW(A), add A → α to M[A, b]

LL(1) Conflicts

CauseExampleFix
Left recursionE → E + T | TRemove: E → T E'
Common prefixA → x Y | x ZLeft factor: A → x A'
 
Left factoring example:
  Before: Stmt -> if (E) Stmt | if (E) Stmt else Stmt
  After:  Stmt -> if (E) Stmt Stmt'
          Stmt' -> else Stmt | epsilon

  • Compute FOLLOW sets using all 4 rules
  • Handle nullable productions in FIRST correctly
  • Build an LL(1) parse table from FIRST/FOLLOW
  • Detect LL(1) conflicts and apply left factoring

Recursive Descent Parsing

One function per non-terminal. The grammar IS the code.

Every non-terminal becomes a function. Every production becomes a branch. You are not writing a parser -- you are transcribing a grammar.

Translation Rules

Grammar FormCode Pattern
A = X Y Z (sequence)Call X(); Y(); Z() in order
A = X | Y (alternation)if tok in FIRST(X) then X() else Y()
A = { X } (repetition)while tok in FIRST(X): X()
Terminal tmatch(t)
Non-terminal BCall B()

match() Function

match(expected):
    if tok == expected:
        tok = next_token()    // ADVANCE
    else:
        error()               // VERIFY

Does two things: verify the current token, then advance. Forgetting either is a bug.

Full Example

Grammar: ex = term { op term }, op = + | -, term = ( ex ) | id

op():
    if tok == '+':  match('+')
    elif tok == '-': match('-')
    else: error()

term():
    if tok == '(':
        match('(')
        ex()
        match(')')
    elif tok == 'id':
        match('id')
    else: error()

ex():
    term()
    while tok == '+' or tok == '-':  // {op term} = while loop
        op()
        term()

Operator Precedence via Grammar Structure

ex   = term { (+ | -) term }    // LOW precedence (processed last)
term = factor { * factor }      // HIGH precedence (processed first)
factor = ( ex ) | id | num      // BASE (atoms)

Higher-precedence operators go deeper in the grammar. term() completes before ex() processes the next +, so multiplication binds tighter.

Error Handling

  1. Failed matches: match() calls error when token is wrong
  2. Default branches: Every function has an else: error()
  3. Return codes: Functions return 1=OK, 0=error; caller propagates
Left recursion = infinite loop. ex() calling ex() immediately without consuming tokens overflows the stack. Always fix the grammar first.
2 skills that cover 80%: (1) Translate grammar to parser code mechanically. (2) Trace parser execution step by step for a given input.

Exam Checklist

  • Write parser functions from any grammar
  • Know when to use if/else vs while
  • Handle lookahead (peek) vs matching (consume)
  • Trace parser execution for a given input
  • Explain how grammar structure enforces precedence

Parse Trees & Action Routines

Two stacks + makebintree() = the entire tree-building mechanism

Parse Tree Structure

Expression: a = b + c List form: (= a (+ b c)) Tree form: = / \ a + / \ b c

Cons Cell Operations

OperationReturnsDescription
isatom(a)boolIs this a leaf node?
head(a)first elementLeft child of dot-pair
tail(a)rest of listRight child of dot-pair
cons(h, t)new pairCreate cell with head=h, tail=t
newatom(x)new atomCreate leaf with value x

Two-Stack Strategy

StackStoresOperations
Argument stack (ystack)Operands and subtreespush(arg), pop()
Operator stack (zstack)Operators (+, -, *)pushop(op), popop()

makebintree()

makebintree():
    b  = pop()        // right operand
    a  = pop()        // left operand
    op = popop()      // operator
    c  = cons(op, cons(a, cons(b, NIL)))  // build (op a b)
    push(c)           // push subtree back

Parser with Action Routines

op():   pushop(tok); match(tok)

term():
    if tok == '(':
        match('('); ex(); match(')')
        makebintree()       // bind inner expression FIRST
    elif tok == id:
        push(tok); match(id)

ex():
    term()
    if tok == '+' or tok == '-':
        op(); term(); makebintree()

Trace: input (x + y)

#ActionArg StackOp Stack
1match '('[][]
2push(x), match(id)[x][]
3pushop(+), match('+')[x][+]
4push(y), match(id)[x, y][+]
5match ')'[x, y][+]
6makebintree()[(+ x y)][]
Parenthesized expressions: makebintree() is called inside term() immediately after matching ). This ensures inner operations are bound into a subtree BEFORE the outer expression continues. That is how (a+b)*c correctly groups a+b first.
The entire topic reduces to one skill: trace the argument stack and operator stack step by step through an expression. Master that trace and you master this chapter.

Code Generation & S-Code

The code generator IS an interpreter that writes instead of executes

S-Code Virtual Machine

Stack-based, zero-address VM. 32-bit instructions: 24-bit argument | 8-bit opcode. 36 total instructions.

Instruction Reference

CategoryInstructions
Dataget.N (push local), put.N (store local), ld.x (push global), st.x (store global), lit.N (push constant), ldx/stx (array)
Arithmeticadd, sub, mul, div, mod, inc, dec
Logicband, bor, bxor, shl, shr, not
Compareeq, ne, lt, le, gt, ge
Controljf.L (jump if false), jt.L (jump if true), jmp.L (unconditional)
Functionfun.x (header), call.f (call), ret.y (return), sys.N (syscall)

The 5 Code Generation Patterns

Pattern 1: Arithmetic (locals)

Source: a = b + c  (a=#1, b=#2, c=#3)
S-code: get.2  get.3  add  put.1

Pattern 2: Arithmetic (globals)

Source: a = b + c  (globals)
S-code: ld.b  ld.c  add  st.a

Pattern 3: if-then-else

Source: if (a == b) c = 1 else c = 2
S-code: get.1  get.2  eq  jf.xx
        lit.1  put.3  jmp.yy
   :xx  lit.2  put.3
   :yy

Pattern 4: Function call

Source: mysum(3, 4)
S-code: lit.3  lit.4  call.mysum

Pattern 5: Recursive function

Source: sum(n,m) { if(n==m) return m; else return n+sum(n+1,m); }
S-code:
:sum    fun.1
        get.1  get.2  eq  jf.L18
        get.2  ret.3
        jmp.L26
:L18    get.1  get.1  lit.1  add  get.2
        call.sum  add  ret.3
:L26    ret.3

Patching Technique

Forward jumps: When emitting jf, you don't know the target yet. Protocol: (1) emit jf.0 placeholder, (2) save ads = CP-1, (3) generate body, (4) patch(ads, CP-ads) to fill correct offset.

eval() Pseudocode

eval(e):
    switch type(e):
        PLUS:  eval(left); eval(right); out(ADD)
        IF:    eval(cond); out(JF,0); ads=CP-1
               eval(body); patch(ads, CP-ads)
        CALL:  for arg in args: eval(arg)
               out(CALL, func)
Master these 5 patterns and you can generate S-code for anything on the exam. Every question is a composition: arithmetic uses Pattern 1/2, branching uses Pattern 3, functions use Pattern 4, recursion is Pattern 5.

  • Generate if-then-else with correct jf/jmp labels
  • Apply patching: emit placeholder, save ads, generate body, patch
  • Generate function calls (push args, call)
  • Trace evaluation stack for S-code sequence

Activation Records & Procedure Linkage

How function calls actually work at the machine level

Memory Layout

+------------------+ | Code | fixed size, read-only +------------------+ | Static Data | globals, fixed size +------------------+ | Stack | | grows DOWN (activation records) | v | | | | ^ | | Heap | | grows UP (dynamic allocation) +------------------+

AR Structure

+------------------------+ | Return value | +------------------------+ | Actual parameters | +------------------------+ | Control link | -> caller's AR +------------------------+ | Access link | -> lexically enclosing AR +------------------------+ | Saved machine status | regs, return addr +------------------------+ | Local data | +------------------------+ | Temporaries | +------------------------+

x86-64 Concrete Offsets

OffsetContents
8n+8(%rbp) to 16(%rbp)Arguments (arg n to arg 1)
8(%rbp)Return address
0(%rbp)Old frame pointer (saved %rbp)
-8(%rbp) to -8m(%rsp)Local variables

4 Phases of Procedure Linkage

Phase 1: Pre-call (CALLER)

Allocate callee AR, store args, save return addr, save caller-save regs, set access link, jump to prolog.

Phase 2: Prolog (CALLEE)

Save old FP, set new FP, save callee-save regs, allocate locals, initialize.

Phase 3: Epilog (CALLEE)

Store return value, restore callee-save regs, restore caller FP, load return addr, jump back.

Phase 4: Post-return (CALLER)

Copy return value, free callee AR, restore caller-save regs, continue.

Register Saving

TypeWho savesWhy
Caller-saveCaller (before call)Caller knows which values are LIVE
Callee-saveCallee (in prolog)Callee knows which regs it will use

Access Link vs. Control Link

Access link follows the SOURCE CODE nesting (lexical). Control link follows the CALL HISTORY (dynamic). They are the same only when calling a directly enclosing function.
Control LinkAccess Link
Points toCaller's ARLexically enclosing AR

Static Coordinates <level, offset>

Recursive fact(3) Example

+-------------------+ | fact(3) n=3 | control -> caller +-------------------+ | fact(2) n=2 | control -> fact(3) +-------------------+ | fact(1) n=1 | control -> fact(2) (base case, returns 1) +-------------------+ Unwind: fact(1)=1, fact(2)=2*1=2, fact(3)=3*2=6
Draw the AR, trace the 4 phases, know access vs control link. These three skills cover nearly every exam question on this topic.

Garbage Collection

Automatic memory management -- when the runtime cleans up the heap

Roots are in activation records. Follow pointers from roots = alive. Everything else = garbage.

Mark-and-Sweep: 4 Phases

#PhaseWhat Happens
1MarkWalk all active ARs. Follow pointers to heap. Mark reached blocks as "alive" via header marker bit.
2Sweep & CopyScan entire heap using block sizes in headers. Copy live blocks to temp space. Record old→new in address map.
3Free & ReturnFree entire original heap. Move live blocks back (compacted). Update address map.
4Update AddressesWalk all ARs again. Use address map to patch every pointer to new locations.

Advantages: Handles circular references. No overhead during normal execution.
Disadvantages: Long stop-the-world pause. Unpredictable latency. Needs extra copy space.

Reference Counting

Each heap object has a counter. Compiler inserts code to increment on new reference, decrement when reference dies. Free when counter = 0.

Advantages: Predictable runtime. Immediate reclamation. No scanning.
Disadvantages: Expensive updates every assignment/scope change. Compiler must be absolutely correct. Difficult edge cases.

Comparison

AspectMark-and-SweepReference Counting
Circular refsHandles correctlyCANNOT handle
Runtime overheadLow (batch)High (every pointer update)
Pause timeLong, unpredictableShort, predictable
Correctness burdenOn GC runtimeOn compiler
ReclamationDelayed (batch)Immediate
CompactionYes (copy phase)No (leaves holes)

Generational GC

Key observation: "Memory that is new is more likely to be used and freed." Most objects die young.

GenerationContainsCollected
Young (nursery)Newly allocatedFrequently (minor GC, fast)
Old (tenured)Survived previous cyclesRarely (major GC, slow)

Objects promoted from young to old after surviving a GC cycle. Young gen is small = quick to collect. Dramatically reduces pause times.

3 things to know: (1) The 4 M&S phases by name and action. (2) RC trade-offs (no circular refs, immediate, but compiler burden). (3) Why generational helps (most die young, small young gen = fast minor GC).
